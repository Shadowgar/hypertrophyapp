- Substitution rule parsing and equipment-mismatch execution now also live in `packages/core-engine/core_engine/rules_runtime.py`; `packages/core-engine/core_engine/scheduler.py` now routes compatible-substitution selection through that shared runtime during week generation, and generated exercises emit a `substitution_decision_trace` describing whether the authored exercise remained compatible or auto-swapped to the first compatible substitute.
- `substitution_rules.repeat_failure_trigger` is now operational on a real runtime path: `POST /workout/{workout_id}/log-set` updates persisted under-target streak state, `GET /workout/today` rehydrates the same stateful outcome, and live workout guidance now emits an optional `substitution_recommendation` once the canonical failed-exposure threshold is reached.

## Immediate Next Execution

1. Start consuming the runtime-backed canonical user-training-state payload in additional decision families instead of repeating ad hoc router-level reads.
2. Keep moving any remaining meaningful router-owned coaching decisions behind `packages/core-engine` interpreters.
3. Require structured decision traces for each migrated decision family.
4. Contain legacy runtime paths so they do not receive new coaching logic.
