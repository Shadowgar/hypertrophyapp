# Generated Plan Remediation Roadmap

## Purpose

Provide a phased execution plan for bringing generated-plan behavior into compliance with canonical doctrine while preserving authored program sovereignty.

## Governance Constraints

- Authored Full Body Phase 1 and Full Body Phase 2 remain separate first-class authored paths.
- Generated-plan runtime must never overwrite or reinterpret authored templates.
- This roadmap begins with docs/governance and then staged runtime work.
- No database migrations are implied by this document.

## Phase 0: Docs And Governance Only

Deliverables:

- doctrine architecture documents for generated-plan family
- exercise metadata model specification
- runtime authority boundary updates
- governance constitution update for authored-vs-generated family separation

Exit criteria:

- doctrine is explicit
- authored/generated boundaries are explicit
- no runtime files changed

## Phase 1: Onboarding Model

Deliverables:

- onboarding decision branch for last-4-weeks training consistency
- deterministic routing rules:
  - complete layoff -> comeback/bridge path
  - partial layoff -> reduced-volume re-entry block
  - consistent training -> normal generated-plan flow

Exit criteria:

- onboarding outcomes traceable
- re-entry branch selectable by deterministic rules

## Phase 2: Exercise Metadata Expansion

Deliverables:

- exercise metadata schema implementation
- metadata population and QA checklist
- collision and substitution tags available for decision modules

Exit criteria:

- required metadata coverage threshold met
- metadata integrity checks pass

## Phase 3: Split Selection Engine

Deliverables:

- split selection policy using days, preference, recovery, training status, and time constraints
- deterministic split recommendation outputs with decision traces

Exit criteria:

- split selection explainable from canonical inputs
- no authored-path regressions

## Phase 4: Base Generated Hypertrophy Plan

Deliverables:

- stable-in-block exercise allocation policy
- progression logic emphasizing logbook trend and RPE/RIR feedback
- substitution engine constrained by training-effect preservation rules

Exit criteria:

- generated base plans respect block-stability doctrine
- progressions and substitutions are trace-auditable

## Phase 5: Time-Constrained Essentials-Style Mode

Deliverables:

- 2-5 day support
- lower-volume, higher-effort session templates
- ~45 minute session targeting logic
- superset pairing rules for non-interfering patterns
- explicit interference exclusions

Exit criteria:

- mode stays within time envelope targets
- interference checks reliably block invalid pairings

## Phase 6: Specialization Layer

Deliverables:

- single-primary-target default policy
- conditional second-target compatibility checks
- collision-aware fatigue budgeting rules
- specialization cycle spacing rule (no immediate identical repeat)

Exit criteria:

- specialization plans remain recoverable
- collision constraints enforced and trace-visible

## Phase 7: Advanced PPL Phase Architecture

Deliverables:

- 6x PPL support for advanced users
- phase rotation support:
  - base hypertrophy
  - maximum overload
  - supercompensation/high-volume
- advanced-method gating by training age/recovery/preference

Exit criteria:

- advanced methods only applied when eligibility criteria pass
- hypertrophy-first default preserved unless strength specialization is selected

## Phase 8: Recomp/Progress Tracking Integrations

Deliverables:

- progress signal tracking:
  - weekly average bodyweight
  - waist
  - photos
  - optional body-part measurements
- goal labels:
  - fat loss
  - recomposition
  - lean gain
  - strength focus
- protein adequacy field for future coaching
- recovery expectation annotations based on energy-balance context

Exit criteria:

- tracking data available to coaching surfaces
- workout generator behavior remains doctrine-consistent without direct nutrition rewrites

## Cross-Phase Risks

- Metadata incompleteness can undermine substitution and collision quality.
- Overlap/collision logic can become too permissive without conservative defaults.
- Specialization demand may exceed recoverable capacity if fatigue budgets are under-modeled.
- User-facing claims can outpace implemented authority if trace coverage is incomplete.

## Recommended Sequencing Enforcement

- Do not begin Phase 4+ runtime generation changes before Phase 1-3 contracts are validated.
- Require doctrine/trace tests at each phase boundary.
- Run authored-path regression checks at each phase to preserve family separation.
