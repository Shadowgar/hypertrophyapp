# Canonical Program Schema (PDL)

## Purpose
Program Definition Language (PDL) is the deterministic contract between build-time knowledge conversion and runtime plan generation.

## Top-Level Fields
- `id`: unique template identifier (e.g., `full_body_v1`).
- `version`: semantic version for template evolution.
- `split`: `full_body` | `ppl` | `upper_lower`.
- `days_supported`: integer array of supported weekly frequency.
- `deload`: deterministic deload trigger and reduction settings.
- `progression`: progression mode and increment defaults.
- `sessions`: ordered session definitions.

## Session Object
- `name`: human readable title.
- `exercises`: ordered list of exercise blocks.

## Exercise Object
- `id`: stable internal exercise key.
- `primary_exercise_id`: canonical default for the slot (usually equals `id`).
- `name`: display name.
- `sets`: planned work sets count.
- `rep_range`: `[min_reps, max_reps]`.
- `start_weight`: default starting working weight.
- `priority`: `high` or `standard` for missed-day roll-forward logic.
- `movement_pattern` (optional): normalized pattern key for deterministic substitutions.
- `primary_muscles` (optional): array of primary muscles.
- `equipment_tags` (optional): explicit equipment tags (`barbell`, `dumbbell`, `cable`, `machine`, `bodyweight`, ...).
- `substitution_candidates` (optional): ordered alternative exercise ids.
- `notes` (optional): slot-level intent/cues to display during workout execution.
- `video.youtube_url` (optional): external video link shown in runner UI.

## Importer Fallback Rule
- Importers may infer `equipment_tags` from name tokens when explicit tags are absent:
	- `DB`/`D.B.`/`Dumbbell` -> `dumbbell`
	- `BB`/`Barbell` -> `barbell`
	- `Cable` -> `cable`
	- `Machine` -> `machine`
	- `BW`/`Bodyweight` -> `bodyweight`
- Explicit tags in source templates always override this fallback parsing.

## Warmups
Warmups are not hardcoded in templates by default; they are generated by `compute_warmups` from working weight and rounding rules.

## Deload Rules
Template `deload` block defines baseline triggers. Runtime may apply stricter triggers from nutrition phase (e.g., cuts deload earlier).

## Progression Rules
Template `progression` block defines deterministic progression mode, with API/core-engine applying:
- rep-target hit + set completion -> load increase
- underperformance -> hold/decrease
- phase modifier scales aggressiveness

## Versioning
- New templates or rule changes require version bump.
- Existing plans keep template version pinned for reproducibility.
- Migration scripts should map old template ids/fields to new contracts.

## Runtime Validation
- API template loading validates canonical templates against this schema before week generation.
- Invalid templates are rejected with an explicit `422` response and are not used for planning.
